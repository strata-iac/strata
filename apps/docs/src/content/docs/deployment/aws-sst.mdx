Procella can be deployed serverlessly on AWS using SST (Serverless Stack). This approach uses Lambda for the API, Aurora Serverless v2 with Data API for the database, S3 for blob storage, and CloudFront for the static UI. Zero servers to manage, scales to zero, ~$55-65/month base cost.

**Official docs**: [sst.dev/docs](https://sst.dev/docs/)

---

## Prerequisites

- AWS account with credentials configured
- Node.js 18+ and Bun
- SST CLI (`npx sst`)

---

## Infrastructure Overview

The SST configuration is split into modular infrastructure components:

| File | Purpose |
|---|---|
| `sst.config.ts` | Root configuration, stacks assembly |
| `infra/database.ts` | Shared VPC, Aurora Serverless v2 with RDS Proxy (0.5-16 ACU) |
| `infra/storage.ts` | S3 bucket for Pulumi state blobs |
| `infra/secrets.ts` | SST secrets (encryption key, auth tokens, Descope keys) |
| `infra/api.ts` | Compiled Bun binary on Lambda (`provided.al2023`, handler at `apps/server/src/lambda-bootstrap.ts`) |
| `infra/gc.ts` | GC worker Lambda on 1-minute EventBridge schedule (same Bun binary approach) |
| `infra/site.ts` | React UI dashboard (StaticSite with CloudFront) |
| `infra/descope.ts` | Descope auth provisioning (optional) |

<Aside type="note">
The `dev` property in `infra/database.ts` overrides Aurora with local Docker PostgreSQL during `sst dev`. This keeps iteration fast and avoids AWS costs during development.
</Aside>

## Set Secrets

Set required secrets before deploying:

```bash
npx sst secret set ProcellaEncryptionKey "$(openssl rand -hex 32)"
npx sst secret set ProcellaDevAuthToken "your-token-here"
```

For Descope authentication mode (production only — the Descope project ID is provisioned automatically via `infra/descope.ts`):

```bash
npx sst secret set ProcellaDescopeManagementKey "..."
```

<Aside type="tip">
Secrets are encrypted and stored in AWS Systems Manager Parameter Store. They are automatically injected as environment variables into your Lambda functions.
</Aside>

## Local Development

SST provides a local development mode that uses your local Docker PostgreSQL instead of Aurora:

```bash
# Start local PostgreSQL
docker compose up -d postgres

# Run SST dev mode
npx sst dev
```

In dev mode:
- The database uses local Docker PostgreSQL (skips Aurora deployment)
- Lambda functions run locally with hot reloading via Live Lambda Development
- The React UI is served by Vite dev server with HMR
- All AWS resources are stubbed or use local equivalents

<Aside type="note">
The `dev` override in `infra/database.ts` configures SST to use your local Docker PostgreSQL connection details during `sst dev`, so Aurora isn't provisioned in this mode. This keeps iteration fast and avoids AWS costs during development.
</Aside>

## Deploy

Deploy to a specific stage (environment):

```bash
npx sst deploy --stage production
```

SST uses stage-based deployment:
- `dev` — Local development with live Lambda
- `staging` — Pre-production testing
- `production` — Production workload

Each stage gets isolated resources. View outputs after deployment:

```bash
npx sst state --stage production
```

This shows the API URL, CloudFront distribution URL, and other stack outputs.

## Database and Migrations

Procella uses Aurora Serverless v2 with RDS Proxy for connection pooling. Lambda functions connect directly via `Bun.sql` (the same driver used in local development and Docker deployments), running inside the VPC for Aurora connectivity.

**Key benefits of direct PG via RDS Proxy:**
- Same `Bun.sql` driver as local dev (no separate Data API driver)
- RDS Proxy handles Lambda connection pooling automatically
- No per-query HTTP overhead (direct TCP connections)
- `PROCELLA_DATABASE_URL` is the only config needed, everywhere

Run database migrations:

```bash
npx sst shell --stage production -- bun apps/server/src/index.ts --migrate
```

<Aside type="tip">
Lambda functions are compiled Bun binaries (`bun build --compile --bytecode --minify`) deployed as custom runtimes (`provided.al2023`). This eliminates all Node.js/esbuild bundling and runs the exact same Bun runtime as local development.
</Aside>

## Scaling

**Aurora Serverless v2 scaling:**

| Setting | Default | Description |
|---|---|---|
| min | 0.5 ACU | Minimum capacity (can set to 0 for dev) |
| max | 16 ACU | Maximum capacity |

Adjust in `infra/database.ts`:

```typescript
scaling: {
  min: "0.5 ACU",
  max: "16 ACU"
}
```

**Lambda scaling:**
- Automatic concurrency scaling
- No configuration needed
- Pay per invocation

**StaticSite (CloudFront):**
- Global CDN with edge caching
- Automatic scaling

<Aside type="tip">
Set `scaling.min` to `"0 ACU"` in dev/staging to pause the database when idle. Saves ~$43/month per non-production stage. Resume takes ~15 seconds on first connection.
</Aside>

## Environment Variables

SST automatically maps `PROCELLA_*` environment variables via `infra/api.ts`. The following are set automatically:

| Variable | Source |
|---|---|
| `PROCELLA_DATABASE_URL` | Constructed from Aurora RDS Proxy host/port/credentials in `infra/database.ts` |
| `PROCELLA_BLOB_BACKEND` | Set to `s3` automatically |
| `PROCELLA_BLOB_S3_BUCKET` | S3 bucket name from `infra/storage.ts` |

All other `PROCELLA_*` config works unchanged. See the [configuration reference](../getting-started/configuration.md) for the full list.

## GC Worker

The GC worker runs as a separate Lambda function triggered by an EventBridge cron schedule (1 minute). It cleans up stale updates:

- Running updates with expired leases
- Not-started updates older than 1 hour
- Requested updates that never started

The GC worker is defined in `infra/gc.ts` and uses the compiled Bun binary at `apps/server/src/gc-bootstrap.ts`. It runs independently from the API Lambda to ensure cleanup continues even if the API is under heavy load.

## Removing

To tear down a stage:

```bash
npx sst remove --stage <stage>
```

<Aside type="caution">
Production stages use `removal: "retain"` — Aurora databases and S3 buckets are NOT deleted on `sst remove`. This prevents accidental data loss. Delete them manually in the AWS console if needed.
</Aside>

---

## Additional Notes

**Compiled Bun binaries:** Lambda functions use `bun build --compile --bytecode --minify` to create self-contained executables deployed as `provided.al2023` custom runtimes. No Node.js, no esbuild, no runtime layers.

**VPC configuration:** Both Aurora and Lambda functions run inside the VPC. Lambda connects to Aurora through RDS Proxy for connection pooling. The VPC uses fck-nat for cost-effective NAT instead of AWS NAT Gateway.

**Production protection:** The production stage has `protect: true` set, preventing accidental deletion via `sst remove`.
